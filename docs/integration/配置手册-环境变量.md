# SQLBot-cx `.env` 配置手册

本文档是 `H:\light\project\SQLBot-cx\.env` 文件的权威字段说明。每个配置按"作用 / 取值含义 / 调整建议 / 故障排查信号"组织。

> 新增配置时，请同步更新本文档。敏感值（token / 账号 / 密码 / 租户 ID / 内部 IP）在文档示例中统一使用占位符 `<...>`。

---

## 一、基础项

| 字段 | 典型值 | 作用 |
|---|---|---|
| `PROJECT_NAME` | `SQLBot` | 项目名，用于日志与 OpenAPI 标题 |
| `BACKEND_CORS_ORIGINS` | `http://localhost,http://localhost:5173,…` | 后端允许的跨域来源；**生产新加前端域名必须追加进来**，否则浏览器请求被 CORS 拦 |
| `SECRET_KEY` | 64 位 hex | 本地 JWT 签名密钥（仅本地管理员双通道用到）；**不要重生成**，已签发的本地 token 会全部失效 |
| `DEFAULT_PWD` | `SQLBot@123456` | 新建本地用户的默认密码；**上线前必须改** |
| `LOG_LEVEL` | `DEBUG` / `INFO` / `WARNING` / `ERROR` | 生产建议 `INFO` |
| `SQL_DEBUG` | `False` | `True` 时打印 SQLAlchemy 全部 SQL，性能与日志噪声大 |
| `CACHE_TYPE` | `memory` / `redis` / `None` | 影响 SQLBot 内部 `sqlbot_cache`（与中台 token Redis 是两套） |

## 四、PLATFORM_AUTH_* —— 数据中台统一认证（核心）

### 4.1 三总开关（决定整体行为）

| 字段 | 典型值 | 含义 |
|---|---|---|
| `PLATFORM_AUTH_ENABLED` | `true` | 总开关。打开后，请求带 Bearer token 时会先尝试用中台 Redis 解析 |
| `PLATFORM_AUTH_STRICT_MODE` | `true` | **严格模式**：所有 Bearer token 必须能在中台 Redis 找到，不允许回退本地 JWT。**生产推荐 `true`** |
| `PLATFORM_AUTH_LOGIN_PROXY_ENABLED` | `true` | 打开后 `/api/v1/login/access-token` 不再走本地账号验证，而是把账号密码代理给中台 |

**模式组合速查：**

| ENABLED | STRICT | PROXY | 行为 |
|:---:|:---:|:---:|---|
| ✓ | ✓ | ✓ | **推荐生产**：登录/鉴权全部走中台；本地用户登不上（admin 白名单除外） |
| ✓ | ✗ | ✓ | 兼容：登录走中台；非 chat 路由 token 可能回退本地 JWT，**易踩 `Not enough segments`** |
| ✗ | – | – | 完全本地：忽略中台，纯 SQLBot 自身 |

### 4.2 登录代理相关

| 字段 | 典型值 | 含义 |
|---|---|---|
| `PLATFORM_AUTH_BASE_URL` | `http://<platform_host>:<port>` | 中台 API 基址（无尾斜杠） |
| `PLATFORM_AUTH_LOGIN_PATH` | `/admin-api/system/auth/login` | 中台登录接口路径 |
| `PLATFORM_AUTH_REFRESH_PATH` | `/admin-api/system/auth/refresh-token` | refresh 路径；**当前 SQLBot 未真正调用，留作后续接入** |
| `PLATFORM_AUTH_LOGOUT_PATH` | `/admin-api/system/auth/logout` | logout 路径；**当前 SQLBot 未真正调用** |
| `PLATFORM_AUTH_HTTP_TIMEOUT_SECONDS` | `10.0` | 代理调用中台的超时（秒） |
| `PLATFORM_AUTH_LOGIN_REMEMBER_ME` | `true` | 透传给中台登录 body 的 `rememberMe` |

### 4.3 租户头处理

中台租户过滤器要求请求带 `tenant-id` header，否则报 **"请求的租户标识未传递"**。

| 字段 | 典型值 | 含义 |
|---|---|---|
| `PLATFORM_AUTH_FIXED_TENANT_NAME` | `<tenant_name>` | 写入登录 body 的 `tenantName` |
| `PLATFORM_AUTH_FIXED_TENANT_ID` | `<tenant_id>` | **直接固定 tenant-id**，优先于查询接口；**生产推荐固定**，避免每次登录多一次 HTTP |
| `PLATFORM_AUTH_TENANT_ID_BY_NAME_PATH` | `/admin-api/system/tenant/get-id-by-name` | 当 `FIXED_TENANT_ID` 留空时，按 `FIXED_TENANT_NAME` 自动查 |
| `PLATFORM_AUTH_TENANT_ID_HEADER` | `tenant-id` | 中台租户头字段名（一般不改） |
| `PLATFORM_AUTH_REQUIRE_TENANT_ID_HEADER` | `true` | `true` 时未解析出 tenant-id 直接报 500，不静默继续 |

### 4.4 Redis token 校验（鉴权热路径）

| 字段 | 典型值 | 含义 |
|---|---|---|
| `PLATFORM_AUTH_REDIS_URL` | `redis://<host>:<port>/<db>` | 中台 Redis 地址；**`/db` 编号必须正确**，db0 常常查不到 token |
| `PLATFORM_AUTH_REDIS_KEY_PREFIX` | `oauth2_access_token:%s` | Redis key 模板，`%s` 替换为 token 字符串 |
| `PLATFORM_AUTH_USERINFO_ACCOUNT_FIELD` | `username` | 从 Redis JSON 的 `userInfo` 子字段中取哪个键作为账号；中台默认 `username` |
| `PLATFORM_AUTH_REQUIRE_USER_TYPE` | `2` | 限制 `userType=2` 才放行（一般"普通用户"=2，管理员=1）；**改造前需核对中台账号的 `userType`** |
| `PLATFORM_AUTH_ACCEPT_AUTHORIZATION_HEADER` | `true` | 允许从 `Authorization` 取 Bearer，不仅是 `X-SQLBOT-TOKEN`；方便 Swagger / curl |

### 4.5 本地管理员双通道

避免开 strict 后失去 SQLBot 本地运维入口。

| 字段 | 典型值 | 含义 |
|---|---|---|
| `PLATFORM_AUTH_ALLOW_LOCAL_ADMIN_LOGIN` | `true` | 总开关 |
| `PLATFORM_AUTH_LOCAL_ADMIN_ACCOUNTS` | `admin` | 逗号分隔的白名单账号；命中时走 SQLBot 本地登录与本地 JWT |

**白名单过宽会削弱中台统一认证目标**，建议仅保留运维类账号。

## 六、调整建议速查

**生产环境推荐组合：**
- `PLATFORM_AUTH_ENABLED=true` + `PLATFORM_AUTH_STRICT_MODE=true` + `PLATFORM_AUTH_LOGIN_PROXY_ENABLED=true`
- `PLATFORM_AUTH_ALLOW_LOCAL_ADMIN_LOGIN=true`（仅 admin）
- `PLATFORM_AUTH_FIXED_TENANT_ID` 直接写死，不依赖 `by-name` 查询
- 需额外确认：
  - Redis 能从 SQLBot 进程/容器访问（若中台 Redis 在另一台机，换 IP 并确认 Redis 监听 `0.0.0.0` + 防火墙放行）
  - 中台普通用户的 `userType` 是否真的是 `2`（在中台 `system_users.user_type` 字段核实）

**调试环境：**
- 不要把 `STRICT_MODE` 设 `false` 来"图省事"，兼容模式下中台随机 token 会被尝试当 JWT 解码，产生误导性 `Not enough segments` 报错
- 调试 token 直接用 `redis-cli -n <db> EXISTS oauth2_access_token:<token>` 验证

## 七、典型故障与对应字段

| 现象 | 检查字段 |
|---|---|
| 登录返回 "请求的租户标识未传递" | `PLATFORM_AUTH_FIXED_TENANT_ID`（是否设置）、`PLATFORM_AUTH_TENANT_ID_HEADER` |
| 业务接口 401 + 日志 `Invalid or expired platform access token` | `PLATFORM_AUTH_REDIS_URL` 的 db 编号、`PLATFORM_AUTH_REDIS_KEY_PREFIX`、token 是否过期 |
| 业务接口 500 报 `Not enough segments` | `PLATFORM_AUTH_STRICT_MODE=false` 且 chat 路由判定走漏；改回 `true` |
| 登录后 admin 进不去后台 | `PLATFORM_AUTH_ALLOW_LOCAL_ADMIN_LOGIN=true`，账号是否在 `PLATFORM_AUTH_LOCAL_ADMIN_ACCOUNTS` |
| 登录 401 触发前端 reload 死循环 | `login.py` 已改为登录失败返回 400；仍出现说明前端缓存旧版本 |
| 跨域被拦 | `BACKEND_CORS_ORIGINS` 漏配前端实际访问域名 |
| Embedding 报模型不存在 | `LOCAL_MODEL_PATH` 路径不存在；是**已知隐患** |
| 图表无法显示 | `SERVER_IMAGE_HOST` 端口与实际后端不一致 |

## 八、联调常用命令

### Redis 验证（脱敏）

```bash
redis-cli -h <redis_host> -p 6379 -n <db> PING
redis-cli -h <redis_host> -p 6379 -n <db> INFO keyspace
redis-cli -h <redis_host> -p 6379 -n <db> --scan --pattern 'oauth2_access_token:*'
redis-cli -h <redis_host> -p 6379 -n <db> EXISTS "oauth2_access_token:<token>"
```

### 登录与鉴权联调

```bash
# 登录取 token
curl --noproxy '*' -i -X POST '<sqlbot_api>/api/v1/login/access-token' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'username=<account>' \
  --data-urlencode 'password=<password>'

# 带 token 访问业务接口
curl --noproxy '*' -i '<sqlbot_api>/api/v1/user/info' \
  -H 'X-SQLBOT-TOKEN: Bearer <access_token>'
```
