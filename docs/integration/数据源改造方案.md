## 1. 目标与场景

把 SQLBot 从"用户手动选择数据源"改为"**统一从数据中台 data 服务取数据源**"，用户视角：

- chat 创建阶段不再被强制绑定 ds（无感）
- 可按 tableId 精确锁定单表（精细化）
- generate_sql 后自动校验表级权限；无权时 SSE 返回申请流程（受控）

**为什么做：**
- 业务方期望"统一产品"——用户不应在中台已经登录的情况下，进 chat 还要再选一次"数据源"
- 中台已是数据治理与权限的事实源头，SQLBot 不应再自建一套"数据源 + 权限"模型
- 表级权限必须在后端 gate，前端隐藏不是安全边界

---

## 2. 架构原则（关键决策）

### 2.1 不复刻、只代理/透传（与主线一一致）

- **不**把中台元数据同步到 SQLBot 本地 `core_datasource` 表
- **不**把中台 DB 连接信息（host / port / username / password）缓存进 SQLBot DB
- **走运行时远程 HTTP 拉取**（复用已有 `AssistantOutDs.get_ds_from_api` 能力）
- 虚拟 assistant 注入到 `request.state.assistant`，让 chat 主流程**完全不改分支**

> **为什么强调"不缓存连接信息"**：codex memory 与外部方案对比后明确——缓存连接串到 SQLBot DB 会引入"密码二次落盘"的安全风险，且中台元数据/权限变更时缓存失效困难。坚持运行时拉取是更干净的边界。

### 2.2 chat 粒度绑定（MVP），未来再演进到 question 粒度

| 粒度 | 优点 | 缺点 | 当前选择 |
|---|---|---|---|
| chat 粒度 | 改动最小，复用 `Chat.datasource` 字段 | 同 chat 跨业务域问数会沿用旧 ds | ✅ MVP |
| question 粒度 | 每个问题独立选 ds，跨域问数稳定 | 需扩 record 数据模型 + 审计 | 🔄 长期 |

### 2.3 与 token 改造（主线一）解耦但接力

主线一已确保 `request.state.current_user` 包含中台身份；主线二在此基础上**再注入** `request.state.assistant` 来驱动远程 ds 拉取。两条主线在 `TokenMiddleware` 串起来，但**对外接口不变**。

## 3. 与需求文档的对应关系

| 需求 | 方案 |
|---|---|
| 跳过数据源选择 | chat/start 不必填 `datasource`；question 阶段触发 `select_datasource` |
| 数据源自动匹配 | 复用 `get_assistant_ds` + LLM 选 ds，ds 列表来源切到中台远程接口 |
| 引导匹配合适 ds | 引入"硬过滤 + table comment 召回 + LLM 重排"策略（详见 §6） |
| 无可用 ds 时 SSE 询问 | 在 `select_datasource` 抛异常的位置改 SSE `datasource_not_found` 消息 |

| 无权表 SSE 申请 | generate_sql 后 + execute_sql 前，**双 Gate** 调中台权限接口；无权时 SSE `permission_denied` |

## 5. 分阶段落地

### Phase 1：跳过数据源选择（不动 DB / 不动中间件）

**范围：**
- 加 `PLATFORM_DATASOURCE_*` 配置占位（**不启用逻辑**）
- 放开 chat/start 必填校验：`CreateChat.datasource` 允许 None、`@require_permissions` 不拦截
- `create_chat` 中 `require_datasource` 分支可绕过
- **前端暂不改**，前端仍可传 ds 走原逻辑

**验收：**
- 前端不传 `datasource` 调 chat/start 成功创建 chat
- 前端传 `datasource` 的老路径行为完全不变
- chat/question 在无 ds chat 上能触发 `select_datasource`（即使当前它仍从本地 `CoreDatasource` 查）

**回滚成本：**极低。仅去掉装饰器与 schema 默认值即可。

### Phase 2：虚拟 assistant 注入（中台 token → 远程 ds）

**范围：**
- `TokenMiddleware.validateToken` 通过中台 token 后：
  - 若 `PLATFORM_DATASOURCE_ENABLED=true`，构造虚拟 `AssistantHeader`
    - `type=1`（落入 `dynamic_ds_types`）
    - `configuration` 塞中台 data 服务 endpoint
    - `certificate` 塞当前 Bearer token（透传到中台 data 服务）
  - 写入 `request.state.assistant`
- `AssistantOutDs.__init__` 加 LRU/TTL 缓存（key: `tenantId:userId:assistantId`，TTL 60s）
- 数据中台 data 服务需提供 `/openapi/sqlbot/datasources/query` 接口

**验收：**
- 中台 token 请求 chat/question，能从中台 data 服务拉 ds 列表
- LLM 能正常从远程 ds 列表里选一个
- 并发请求不打爆中台 data 服务（HTTP 缓存命中）

**回滚成本：**中。注入逻辑收敛在 middleware，关掉 `PLATFORM_DATASOURCE_ENABLED` 即可降级到 Phase 1。

### Phase 3：表级绑定 + 权限 SSE

**范围：**
- `Chat` / `CreateChat` 新增 `table_list: List[str]` 字段（alembic migration）
- 前端 chat/start 可传 tableId 列表
- `llm.py` 在 `choose_table_schema` / `get_table_schema` 调用处注入 `table_list`
- **Gate 1**：`check_sql` 后、`generate_filter` 前，调中台权限接口
- **Gate 2**：`execute_sql` 前，对最终 SQL 二次解析 + 权限复核（防 `generate_filter` / `generate_assistant_filter` 二次改写绕过）
- 无权时 SSE `permission_denied` + `applyUrl`
- `select_datasource` 无候选时改 SSE `datasource_not_found`

**验收：**
- chat 绑定单表后，生成的 SQL 只涉及该表
- 查询无权表触发 SSE 申请流程，且后端确实**不执行 SQL**
- 中台权限服务异常时**默认拒绝**执行（fail-closed）
- 双 Gate 都覆盖：模拟 LLM 输出 tables 与 SQL parser 解析结果不一致时，能拦下越权

**回滚成本：**高。涉及 DB schema 变更，需要回滚 migration。

## 6. 数据源匹配策略（Phase 2/3 详化）

> 当用户身份能拉到多个候选数据源时，需要自动选一个最合适的喂给 LLM 生成 SQL。这一节说明匹配策略。
>
> 来源：吸收自 `H:\light\llm-wiki\raw\sqlbot_build_plan\相关说明\SQLBot 数据源自动选择改造方案.md`，参数与流水线沿用其建议。

### 6.1 七步策略流水线

```
第 0 步：硬过滤
  仅保留 online=true、用户可访问、Assistant 允许、租户内的数据源

第 1 步：目录上下文加权
  若前端传 currentCatalogId / currentCatalogPath，同目录候选加权

第 2 步：table comment 召回
  用 table.name + displayName + comment 召回 topK tables，按 datasource 聚合分数

第 3 步：field comment 补充
  字段名 + 注释召回指标/维度，反推 table → datasource

第 4 步：datasource description 召回
  用数据源描述补充业务域级别判断

第 5 步：用户默认数据源兜底（仅加权，不覆盖语义匹配）

第 6 步：LLM 重排
  对 top3 datasource 输入简短 schema 摘要，LLM 给出选择理由

第 7 步：阈值判断
  最高分低于 minScore，或 top1/top2 差距低于 tieDelta，返回前端引导
```

### 6.2 推荐参数

| 参数 | 建议值 | 说明 |
|---|---|---|
| `topKDatasource` | 5 | 进入 LLM 重排的数据源数量上限 |
| `topKTable` | 30 | 单次问题进入 schema prompt 的表数量上限 |
| `topKField` | 120 | 字段召回上限 |
| `minScore` | 0.62 | 低于此分认为没有可靠数据源 |
| `tieDelta` | 0.08 | top1/top2 差距小于此值认为不确定 |

### 6.3 综合评分公式（建议初版）

```
final_score = 0.20 * datasource_description_score
            + 0.40 * max_table_comment_score
            + 0.20 * max_field_comment_score
            + 0.15 * catalog_context_score
            + 0.05 * is_user_default
```

**说明：** table comment 权重最高，因为问数问题通常直接对应表主题；目录上下文加权解决"用户从销售域菜单进 chat"的场景；默认数据源仅作兜底。

## 7. 权限校验（Phase 3 详化）

### 7.1 为什么必须双 Gate

| 阶段 | Gate | 校验内容 | 作用 |
|---|---|---|---|
| `check_sql` 之后 | **Gate 1** | LLM 返回的 `tables` + SQL parser 解析的 `tables` 合并去重 | 主校验，覆盖 90% 场景 |
| `execute_sql` 之前 | **Gate 2** | 对最终 SQL（可能被 `generate_filter` / `generate_assistant_filter` / `check_save_sql` 二次改写）再次 parse + 校验 | 兜底，防止动态 SQL 路径绕过 |

> **为什么不能只信 LLM 返回的 `tables`**：模型可能漏报或写错；尤其在 join、CTE、子查询场景。必须用 SQL parser 二次解析真实表名。
>
> **为什么不能只在 `check_sql` 后校验**：SQLBot 当前在 `check_sql` 之后还有 `generate_filter`、`generate_assistant_filter`、`check_save_sql` 等改写路径，这些可能会注入新表或动态片段。

### 7.2 失败默认拒绝（fail-closed）

中台权限服务超时、5xx、连接错误时，**默认拒绝执行 SQL**，不能因服务不稳定就放行。

### 7.3 SSE 事件协议

| 事件 type | 触发条件 | payload 示例 |
|---|---|---|
| `datasource_not_found` | 无候选 ds，或 top1 < minScore | `{candidates: [...], reason: "no_candidate_reach_threshold"}` |
| `datasource_changed` | history ds 与当前问题不匹配，重新选择（Phase 3+ 增强） | `{from: "销售数仓", to: "库存数仓", reason: "..."}` |
| `permission_denied` | Gate 1 / Gate 2 校验失败 | `{deniedTables: [...], applyUrl: "..."}` |
| `permission_apply_required` | 用户主动触发申请流程 | `{tableId, catalogPath}` |

## 9. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 中台 data 服务 SLA 低于 chat/question 期望 | 短 TTL 缓存（60s）+ 超时降级 + fail-closed |
| `AssistantOutDs.__init__` 每次同步 HTTP | Phase 2 加 LRU/TTL 缓存（现 cache 装饰器是注释掉的） |
| `tenantId → oid` 映射后，本地 `_ds.oid != current_user.oid` 报错 | 统一走远程 ds 即可绕过；仅当保留本地 ds 兜底时才需要修正 |
| 表名重复（不同 ds 同名表） | 表级绑定使用 (ds_id, table_name) 组合，不只是 table_name |
| chat 中切换 ds | 默认 chat 粒度绑定；演进到 question 粒度是独立项目 |
| 前端适配未同步 | Phase 1 向后兼容；Phase 2 起需要前端配合 |
| LLM 漏报 SQL 涉及表 → 越权 | Gate 1/Gate 2 都用 SQL parser 二次解析，不只信 LLM |
| 中台 schema 过大导致 prompt 超长 | 实施 §6 的 topK 召回，只把 topK tables / fields 传给 LLM |
