# 数据源改造方案（主线二）

> 本文是 SQLBot-cx 数据源改造的**现行方案与实施计划**。需求源文档：`H:\light\data\SQLBot集成修改\需求\SQLBOT数据源改造&权限控制的说明 (1).md` 与 `后端改造方案.md`。

## 1. 目标

把 SQLBot 从"用户手动选择数据源"改为"**统一从数据中台 data 服务取数据源**"，用户视角：
- chat 创建阶段不再被强制绑定 ds
- 可按 tableId 精确锁定单表
- generate_sql 后自动校验表级权限；无权时 SSE 返回申请流程

## 2. 架构决策

### 2.1 遵循主线一的"不复刻、只代理/透传"原则

- **不**把中台元数据同步到 SQLBot 本地 `core_datasource` 表
- **走远程 HTTP 拉取**（复用 `AssistantOutDs.get_ds_from_api` 现成能力）
- 虚拟 assistant 注入到 `request.state.assistant`，让 chat 主流程不改分支

### 2.2 与需求文档的对应关系

| 需求 | 方案 |
|---|---|
| 跳过数据源选择 | chat/start 不必填 `datasource`；question 阶段触发 `select_datasource` |
| 数据源自动匹配 | 复用 `get_assistant_ds` + 现有 LLM 选 ds 逻辑，只是 ds 列表来源改为中台远程接口 |
| 无可用 ds 时 SSE 询问 | 在 `select_datasource` 抛异常的位置改 SSE 消息 |
| 限制单表 | chat 粒度绑 tableId，转发到 `get_table_schema(..., table_list=[...])`（**现成参数**） |
| 无权表 SSE 申请 | `generate_sql` 后、execute 前，调中台权限接口；无权时 SSE `{"type":"permission_required", "tables":[...]}` |

## 4. 分阶段落地

### Phase 1：跳过数据源选择（不动 DB / 不动中间件）

**范围：**
- 加 `PLATFORM_DATASOURCE_*` 配置占位（不启用逻辑）
- 放开 chat/start 必填校验：`CreateChat.datasource` 允许 None、`@require_permissions` 不拦截
- `create_chat` 中 `require_datasource` 分支可绕过
- **前端暂不改**，前端仍可传 ds 走原逻辑

**验收：**
- 前端不传 `datasource` 调 chat/start 成功创建 chat
- 前端传 `datasource` 的老路径行为不变
- chat/question 能触发 `select_datasource`（即使当前它仍从本地 `CoreDatasource` 查）

### Phase 2：虚拟 assistant 注入（中台 token → 远程 ds）

**范围：**
- `TokenMiddleware.validateToken` 通过中台 token 后：
  - 若 `PLATFORM_DATASOURCE_ENABLED=true`，构造虚拟 `AssistantHeader`（type=1，configuration 塞中台 data 服务 endpoint，certificate 塞当前 Bearer token）
  - 写入 `request.state.assistant`
- `AssistantOutDs` 加缓存（构造函数当前每次同步拉 HTTP，并发场景会爆）
- 数据中台 data 服务需提供接口（协议按需求文档 JSON 格式）

**验收：**
- 中台 token 请求 chat/question，能从中台 data 服务拉 ds 列表
- LLM 能正常从远程 ds 列表里选一个
- 并发请求不打爆中台 data 服务（HTTP 缓存命中）

### Phase 3：表级绑定 + 权限 SSE

**范围：**
- `Chat` / `CreateChat` 新增 `table_list: List[str]` 字段（加 alembic migration）
- 前端 chat/start 可传 tableId 列表
- `llm.py` 在 `choose_table_schema` / `get_table_schema` 调用处注入 `table_list`
- `check_sql` 拿到 SQL 里涉及的 tables 后，调中台权限接口
- 无权时 SSE `yield {"type": "permission_required", "tables": [...], "category": ...}`
- `select_datasource` 无可用 ds 时改 SSE `{"type": "datasource_missing", "action_hint": "apply"}`

**验收：**
- chat 绑定单表后，生成的 SQL 只涉及该表
- 查询无权表触发 SSE 申请流程
- 申请接口对接中台"申请表权限"流程

## 5. 新增配置（Phase 2 启用）

```env
# 数据中台 data 服务（数据源元数据来源）
PLATFORM_DATASOURCE_ENABLED=false
PLATFORM_DATASOURCE_BASE_URL=<platform_data_base_url>
PLATFORM_DATASOURCE_LIST_PATH=/data-api/datasource/list
PLATFORM_DATASOURCE_HTTP_TIMEOUT_SECONDS=10.0
PLATFORM_DATASOURCE_CACHE_TTL_SECONDS=60
# 权限校验
PLATFORM_PERMISSION_CHECK_PATH=/data-api/permission/check
PLATFORM_PERMISSION_APPLY_PATH=/data-api/permission/apply
```

具体字段在 Phase 2 实施时最终确定。

## 6. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 中台 data 服务 SLA 低于 chat/question 期望 | 短 TTL 缓存 + 超时降级 |
| `AssistantOutDs.__init__` 每次同步 HTTP | Phase 2 加 LRU/TTL 缓存（现缓存装饰器是注释掉的） |
| `tenantId → oid` 映射后，本地 `_ds.oid != current_user.oid` 报错 | 统一走远程 ds 即可绕过；仅当保留本地 ds 兜底时才需要修正 |
| 表名重复（不同 ds 同名表） | 表级绑定要用 (ds_id, table_name) 组合，不只是 table_name |
| chat 中切换 ds | 本方案默认 chat 粒度绑定；未来改为 question 粒度是独立项目 |
| 前端适配未同步 | Phase 1 向后兼容；Phase 2 起需要前端配合 |

## 8. 与历史会话的差异

相比 2026-02~03 期间 codex 里讨论过的"数据源改造"草案，本方案的差异：

- 明确分三阶段，**Phase 1 不动 DB 不动中间件**，先跑通"无 ds chat"
- 虚拟 assistant 注入时机定在 `TokenMiddleware`（而非 chat 层），让 chat 主流程完全不改分支
- 复用现有 `table_list` 参数，不重新发明过滤机制
