# 认证体系总览

> 本文是 SQLBot-cx 接入数据中台**认证改造（主线一）**的现行说明。归档的过程文档在 [`../archive/`](../archive/)。

## 1. 一句话说明

SQLBot 的认证从"只认本地 JWT"扩展为"识别数据中台 access_token（Redis 存储）"，普通用户走中台登录 + 中台 token，本地管理员仍走 SQLBot 本地登录 + 本地 JWT。

## 2. 完整链路

```
用户在 SQLBot 前端输入账号密码
   ↓
SQLBot 后端 /api/v1/login/access-token
   ↓
若账号 ∈ PLATFORM_AUTH_LOCAL_ADMIN_ACCOUNTS
   → 走本地登录（authenticate）→ 签本地 JWT → 返回前端
否则若 PLATFORM_AUTH_LOGIN_PROXY_ENABLED
   → 解析 tenant-id（FIXED_TENANT_ID 或 by-name 查询）
   → 调中台 /admin-api/system/auth/login
   → 中台校验 → 签 accessToken → 写 Redis
   → SQLBot 把 accessToken 透传给前端

前端保存 accessToken
   ↓
后续每个请求带 X-SQLBOT-TOKEN: Bearer <accessToken>
   ↓
SQLBot TokenMiddleware
   ↓
若 PLATFORM_AUTH_ENABLED 且（strict OR chat 路径）
   → 查 Redis oauth2_access_token:<token>
   → 解析 userId / tenantId / userInfo.username
   → 构造 UserInfoDTO 写入 request.state.current_user
否则
   → strict 模式：401（除非账号是本地 admin 白名单）
   → 兼容模式：尝试本地 JWT 解码（中台随机 token 会失败）
   ↓
业务接口正常消费 current_user
```

| `backend/apps/system/middleware/auth.py` | `TokenMiddleware.dispatch` | 请求入口分流：白名单 / askToken / assistantToken / 普通 token |

## 4. 字段映射

中台 Redis JSON → SQLBot `UserInfoDTO`：

| 中台字段 | SQLBot 字段 | 备注 |
|---|---|---|
| `userId` | `id` | 直接 int |
| `userInfo.<PLATFORM_AUTH_USERINFO_ACCOUNT_FIELD>`（默认 `username`） | `account` | 字段名可配置 |
| `userInfo.nickname` | `name` | 缺失时回退 `account` |
| `tenantId`（或 `oid` 兜底） | `oid` | **直接当组织 ID 用** |
| `userType` | 校验字段 | 可用 `PLATFORM_AUTH_REQUIRE_USER_TYPE` 限制 |
| `userInfo` | `email` | 当前用 `<account>@platform.local` 占位 |
| `expiresTime` | 过期校验 | 兼容字符串/Java 毫秒时间戳 |

## 5. 模式决策

| 场景 | ENABLED | STRICT | PROXY | 说明 |
|---|:---:|:---:|:---:|---|
| **生产推荐** | ✓ | ✓ | ✓ | 全程中台；本地 admin 白名单兜底 |
| 中台未就绪 | ✗ | – | – | 完全本地 |
| 灰度调试 | ✓ | ✗ | ✓ | 不推荐：中台随机 token 易被当 JWT 解码 |
| 仅 chat 接入 | ✓ | ✗ | ✗ | 兼容模式只对 `/api/v1/chat/*` 走中台，其他走本地 |

## 6. 仓库版相对早期草稿的加固

`H:\light\data\SQLBot集成修改\` 下的草稿版相比，仓库版多了三块加固：

1. **本地 admin 兜底**：strict 模式下，白名单账号即便走中台失败也能用本地 JWT 登录，避免完全失联（`auth.py:_try_local_admin_user`）
2. **tenant-id 自动解析**：未配 `FIXED_TENANT_ID` 时按 `FIXED_TENANT_NAME` 查中台 `get-id-by-name`（`login.py:_resolve_platform_tenant_id`）
3. **JSON/时间戳容错**：兼容 Java 毫秒时间戳与二次 JSON 编码（`token_resolver.py:_parse_datetime` / `_parse_json_object`）

## 7. 已知遗留

详见 [`集成总览.md`](./集成总览.md) §5。简要：

- Redis-only 校验，无 MySQL 回退
- refresh / logout 代理未做
- 中台角色/部门/scopes 未映射到 SQLBot 权限体系

## 8. 排错速查

详见 [`配置手册-环境变量.md`](./配置手册-环境变量.md) §7「典型故障与对应字段」。

最常见三个：
- `Not enough segments` → strict 没开，中台随机 token 被当 JWT
- `Invalid or expired platform access token` → Redis db 错或 token 过期
- "请求的租户标识未传递" → 登录代理没带 tenant-id 头
