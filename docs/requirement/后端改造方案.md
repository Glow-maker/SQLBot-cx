## question接口
/chat/question接口触发用户单次问题，并以EventStream的相应的形式返回给前端。

**接口基本信息**

接口路径 : POST /chat/question

返回类型 : Server-Sent Events (SSE) 流式响应

媒体类型 : text/event-stream

**返回格式概述**

该接口使用 Server-Sent Events (SSE) 协议，以流式方式返回数据。每条消息的格式为：

```plain
data: {JSON数据}\n\n
```

**返回的数据类型**

根据不同的处理阶段，接口会返回多种类型的消息：

对话示例

```plain
data: {"type": "id", "id": 123}

data: {"id": 1, "datasource_name": "销售数据库", "ds_type": "postgres"}

data: {"type": "info", "msg": "sql generated"}

data: {"content": "SELECT * FROM sales WHERE date >= NOW() - INTERVAL '7 days'", "type": "sql"}

data: {"content": "execute-success", "type": "sql-data"}

data: {"type": "info", "msg": "chart generated"}

data: {"type": "chart", "content": {"type": "line", "title": "销售趋势", ...}}

data: {"type": "finish"}

```

## 2.RPC方案
方案1：基于通用的rpc的协议和统一注册中心

+ 优势：
    - 注册中心和rpc协议统一，效率高

### 如何解决token兼容性的问题
1. 在chat模块在获取用户时是根据token内容解析的，而非直接查询数据库-》因此我们只需要从head的token字段中仿造数据中台的代码实现token获取逻辑，
    1. 可以通过解析当前中台的token。仿造TokenAuthenticationFilter的代码实现：从redis中获取后设置到state中。
2. chat只需要用户id、用户标识、和组织机构id即可。这部分内容和当前的用户id、用户名、与租户id基本可以对应
3. 由于chat模块中的用户进行均从http请求中恢复，并不存在直接读取数据库数据的情况，因此对于system中的user模块无需改造

## Datasource如何映


当前Datasource实际上对应于一定范围内的数据集，而不是完整的Datasource的概念。

从长期来看：Datasource的权限控制范围不应该有用户来设置，而是根据统一全局元数据管理中心后，在添加用户信息拦截机引导用户申请权限，之后再继续进行NL2SQL的流程。

短期来看：前端在创建chat的过程中直接使用tableid传入后创建chat即可，后续所有的取数范围直接使用指定的表。

要是实现这个目标需要做2间事情

1. 完成sqlbot和数据中台的数据结构映射
2. 限制仅仅查询单个表
